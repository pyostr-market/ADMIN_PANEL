# API Specification — Pricing Engine (Модуль «Прайсы»)

> Base URL: `/api/v1/price`
>
> Все endpoints требуют авторизации (Bearer Token в заголовке `Authorization`).
>
> Все ответы обёрнуты в единый формат: `{ "success": true, "data": ... }` или `{ "success": false, "error": { "code": "...", "message": "..." } }`.

---

## 1. Аутентификация и права доступа

| Permission | Endpoint |
|-----------|----------|
| `pricing_engine:view` | `GET /` , `GET /{id}` |
| `pricing_engine:create` | `POST /` |
| `pricing_engine:update` | `PUT /{id}` |
| `pricing_engine:delete` | `DELETE /{id}` |

> **Примечание для фронтенда:** Токен передаётся в заголовке `Authorization: Bearer <token>`. Если у пользователя нет нужного права — сервер вернёт `403 Forbidden`.

---

## 2. Формат ответов

### Успешный ответ (одиночный объект)
```json
{
  "success": true,
  "data": {
    "id": 42,
    "category": "электроника",
    "supplier": "поставщик1",
    "region": "москва",
    "price_text": "Цена: 1500 руб."
  }
}
```

### Успешный ответ (список с пагинацией)
```json
{
  "success": true,
  "data": {
    "items": [
      {
        "id": 1,
        "category": "электроника",
        "supplier": "поставщик1",
        "region": "москва",
        "price_text": "Цена: 1500 руб."
      },
      {
        "id": 2,
        "category": "одежда",
        "supplier": "поставщик2",
        "region": "спб",
        "price_text": "Цена: 3000 руб."
      }
    ],
    "total": 150
  }
}
```

### Ответ с ошибкой
```json
{
  "success": false,
  "error": {
    "code": "price_not_found",
    "message": "Прайс с id 999 не найден",
    "details": {}
  }
}
```

### HTTP-коды ошибок
| Код | Ситуация |
|-----|----------|
| `400` | Ошибка валидации (длина полей, пустые значения) |
| `401` | Нет авторизации |
| `403` | Нет нужного permission |
| `404` | Прайс не найден |
| `409` | Конфликт: прайс с такой комбинацией category+supplier+region уже существует |
| `422` | Нарушение внешних ключей (несуществующий справочник) |

---

## 3. Схемы данных

### PriceCreateRequest — тело запроса для создания
| Поле | Тип | Обязательное | Ограничения | Описание |
|------|-----|-------------|-------------|----------|
| `category` | string | ✅ | 1–100 символов | Название категории |
| `supplier` | string | ✅ | 1–200 символов | Название поставщика |
| `region` | string | ✅ | 1–100 символов | Название региона |
| `price_text` | string | ✅ | ≥ 1 символа | Текст прайса (цена, описание) |

> **Важно:** Если категория, поставщик или регион не существуют в системе, они будут **автоматически созданы**. Названия автоматически нормализуются к нижнему регистру.

### PriceUpdateRequest — тело запроса для обновления
| Поле | Тип | Обязательное | Ограничения | Описание |
|------|-----|-------------|-------------|----------|
| `category` | string | ❌ | 1–100 символов | Новое название категории |
| `supplier` | string | ❌ | 1–200 символов | Новое название поставщика |
| `region` | string | ❌ | 1–100 символов | Новое название региона |
| `price_text` | string | ❌ | ≥ 1 символа | Новый текст прайса |

> **Важно:** Все поля необязательны. Обновляются только переданные. Оставшиеся `null` поля игнорируются. Если новый справочник не существует — он будет создан.

### PriceResponse — ответ с данными прайса
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int | Уникальный ID прайса |
| `category` | string | Название категории (в нижнем регистре) |
| `supplier` | string | Название поставщика (в нижнем регистре) |
| `region` | string | Название региона (в нижнем регистре) |
| `price_text` | string | Текст прайса |

### PaginatedResponse — ответ списка
| Поле | Тип | Описание |
|------|-----|----------|
| `items` | PriceResponse[] | Массив прайсов текущей страницы |
| `total` | int | Общее количество записей (без пагинации) |

## 4. Endpoints

---

### 4.1 GET `/api/v1/price/` — Список прайсов

**Permission:** `pricing_engine:view`

**Query-параметры:**

| Параметр | Тип | По умолчанию | Ограничения | Описание |
|----------|-----|-------------|-------------|----------|
| `category` | string | — | — | Фильтр по категории |
| `supplier` | string | — | — | Фильтр по поставщику |
| `region` | string | — | — | Фильтр по региону |
| `limit` | int | `100` | 1–1000 | Количество записей на странице |
| `offset` | int | `0` | ≥ 0 | Смещение от начала списка |

> **Примечание для фронтенда:**
> - Все фильтры **независимы** и могут комбинироваться.
> - Фильтрация — **case-insensitive** (сервер приводит к нижнему регистру).
> - Сортировка — по `id` (возрастание).
> - Для пагинации используйте `limit` + `offset`. Общее количество записей — в поле `total`.

**Пример запроса:**
```
GET /api/v1/price/?category=электроника&region=москва&limit=20&offset=0
```

**Пример ответа (200):**
```json
{
  "success": true,
  "data": {
    "items": [
      {
        "id": 1,
        "category": "электроника",
        "supplier": "поставщик1",
        "region": "москва",
        "price_text": "Цена: 1500 руб."
      }
    ],
    "total": 1
  }
}
```

**Пример ответа (пустой список, 200):**
```json
{
  "success": true,
  "data": {
    "items": [],
    "total": 0
  }
}
```

### 4.3 POST `/api/v1/price/` — Создать прайс

**Permission:** `pricing_engine:create`

**Тело запроса:** `PriceCreateRequest`

```json
{
  "category": "Электроника",
  "supplier": "Поставщик1",
  "region": "Москва",
  "price_text": "Цена: 1500 руб."
}
```

**Пример ответа (200):**
```json
{
  "success": true,
  "data": {
    "id": 42,
    "category": "электроника",
    "supplier": "поставщик1",
    "region": "москва",
    "price_text": "Цена: 1500 руб."
  }
}
```

**Пример ответа (409 — уже существует):**
```json
{
  "success": false,
  "error": {
    "code": "price_already_exists",
    "message": "Прайс с такой комбинацией категории, поставщика и региона уже существует",
    "details": {
      "category": "электроника",
      "supplier": "поставщик1",
      "region": "москва"
    }
  }
}
```

**Пример ответа (422 — нарушение внешних ключей):**
```json
{
  "success": false,
  "error": {
    "code": "price_foreign_key_violation",
    "message": "Нарушение целостности данных: ссылка на несуществующий справочник",
    "details": {}
  }
}
```

> **Примечание для фронтенда:**
> - Названия category/supplier/region автоматически нормализуются к нижнему регистру.
> - Если справочник не существует — он создаётся автоматически.
> - Уникальность определяется комбинацией `category + supplier + region`.
> - Поле `price_text` — свободный текст (цена, описание, условия).

### 4.4 PUT `/api/v1/price/{price_id}` — Обновить прайс

**Permission:** `pricing_engine:update`

**Path-параметры:**

| Параметр | Тип | Описание |
|----------|-----|----------|
| `price_id` | int | Уникальный ID прайса |

**Тело запроса:** `PriceUpdateRequest`

```json
{
  "price_text": "Цена: 2000 руб."
}
```

> **Важно:** Передавайте только те поля, которые нужно обновить. Остальные будут сохранены.

**Пример ответа (200):**
```json
{
  "success": true,
  "data": {
    "id": 42,
    "category": "электроника",
    "supplier": "поставщик1",
    "region": "москва",
    "price_text": "Цена: 2000 руб."
  }
}
```

**Пример ответа (404 — не найден):**
```json
{
  "success": false,
  "error": {
    "code": "price_not_found",
    "message": "Прайс с id 999 не найден",
    "details": {}
  }
}
```

**Пример ответа (409 — конфликт уникальности):**
```json
{
  "success": false,
  "error": {
    "code": "price_already_exists",
    "message": "Прайс с такой комбинацией категории, поставщика и региона уже существует",
    "details": {}
  }

```

> **Примечание для фронтенда:**
> - При обновлении category/supplier/region на новые значения — новые справочники создаются автоматически.
> - Если при обновлении возникает дубликат комбинации `category + supplier + region` — вернётся `409`.

---

### 4.5 DELETE `/api/v1/price/{price_id}` — Удалить прайс

**Permission:** `pricing_engine:delete`

**Path-параметры:**

| Параметр | Тип | Описание |
|----------|-----|----------|
| `price_id` | int | Уникальный ID прайса |

**Пример ответа (200):**
```json
{
  "success": true,
  "data": {
    "deleted": true
  }
}
```

**Пример ответа (404 — не найден):**
```json
{
  "success": false,
  "error": {
    "code": "price_not_found",
    "message": "Прайс с id 999 не найден",
    "details": {}
  }
}
```

> **Примечание для фронтенда:**
> - Справочники (category, supplier, region) **не удаляются** каскадно.
> - Если на справочник ещё есть ссылки из других прайсов — он останется.
> - Если ссылка была единственной — ORM-модель удаляет каскадно (`ON DELETE CASCADE`), но это деталь реализации, фронтенд на это не влияет.

## 6. Рекомендации для фронтенда

### 6.1 Страница списка прайсов (таблица)
- Используйте `GET /api/v1/price/` с `limit=50` (или настраиваемым).
- Реализуйте клиентскую пагинацию через `offset` и `total`.
- Фильтры: выпадающие списки или текстовые поля для `category`, `supplier`, `region`.
- При удалении строки — оптимистичное обновление UI, затем `DELETE` запрос.

### 6.2 Страница создания/редактирования прайса
- **Создание:** форма с 4 обязательными полями. При отправке — `POST /api/v1/price/`.
- **Редактирование:** предзаполните форму данными из `GET /api/v1/price/{id}`, отправляйте `PUT /api/v1/price/{id}` только изменённые поля.
- Автодополнение: при вводе категории/поставщика/региона можно предлагать существующие значения через тот же `GET /api/v1/price/?category=...`.

### 6.3 Обработка ошибок
- `409` — показать пользователю «Прайс с такой комбинацией уже существует».
- `422` — показать «Нарушение целостности данных».
- `404` — «Прайс не найден».
- `401`/`403` — редирект на страницу входа или сообщение «Нет прав».

### 6.4 Нормализация данных
- Сервер всегда возвращает `category`, `supplier`, `region` в **нижнем регистре**.
- При отправке запроса регистр не важен — сервер нормализует.

## 7. Архитектурные детали (для контекста)

> Это информационный раздел. Фронтенд не обязан его знать, но он помогает понять поведение API.

- **CQRS:** Чтение (Queries) и запись (Commands) разделены на уровне приложения.
- **DDD:** Бизнес-логика инкапсулирована в доменных агрегатах. Справочники (категории, поставщики, регионы) автоматически создаются при необходимости.
- **Domain Events:** При каждом создании/обновлении/удалении публикуются доменные события.
- **UnitOfWork:** Все операции в рамках одной транзакции.
