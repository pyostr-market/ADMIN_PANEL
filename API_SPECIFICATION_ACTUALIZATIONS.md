# API Specification — Задачи актуализации

> Base URL: `/admin/catalog/actualizations`
>
> Все endpoints требуют авторизации (Bearer Token в заголовке `Authorization`).
>
> Все ответы обёрнуты в единый формат: `{ "success": true, "data": ... }` или `{ "success": false, "error": { "code": "...", "message": "..." } }`.

---

## 1. Аутентификация и права доступа

| Permission | Endpoint |
|-----------|----------|
| `pricing_engine:view` | `GET /admin/catalog/actualizations` , `GET /admin/catalog/actualizations/{id}` |
| `pricing_engine:create` | `POST /admin/catalog/actualizations` |
| `pricing_engine:update` | `PUT /admin/catalog/actualizations/{id}` |
| `pricing_engine:delete` | `DELETE /admin/catalog/actualizations/{id}` |

> **Примечание для фронтенда:** Токен передаётся в заголовке `Authorization: Bearer <token>`. Если у пользователя нет нужного права — сервер вернёт `403 Forbidden`.

---

## 2. Формат ответов

### Успешный ответ (одиночная задача)
```json
{
  "success": true,
  "data": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "user_id": 42,
    "user_full_name": "Иванов Иван Иванович",
    "status": "CREATED"
  }
}
```

### Успешный ответ (список задач с пагинацией)
```json
{
  "success": true,
  "data": {
    "items": [
      {
        "id": "123e4567-e89b-12d3-a456-426614174000",
        "user_id": 42,
        "user_full_name": "Иванов Иван Иванович",
        "status": "COMPLETED"
      },
      {
        "id": "223e4567-e89b-12d3-a456-426614174001",
        "user_id": 42,
        "user_full_name": "Иванов Иван Иванович",
        "status": "CREATED"
      }
    ],
    "total": 15
  }
}
```

### Ответ с ошибкой
```json
{
  "success": false,
  "error": {
    "code": "actualization_task_not_found",
    "message": "Задача актуализации 123e4567-e89b-12d3-a456-426614174000 не найдена",
    "details": {}
  }
}
```

### HTTP-коды ошибок
| Код | Ситуация |
|-----|----------|
| `400` | Ошибка валидации (недопустимый статус) |
| `401` | Нет авторизации |
| `403` | Нет нужного permission |
| `404` | Задача актуализации не найдена |
| `409` | Конфликт (задача уже существует) |
| `500` | Ошибка сервера (не удалось импортировать каталог) |
| `503` | Внешний сервис каталога недоступен |

## 3. Схемы данных

### ActualizationTaskCreateRequest — тело запроса для создания задачи
| Поле | Тип | Обязательное | Описание |
|------|-----|-------------|----------|
| *(нет)* | — | — | Тело запроса пустое. Все данные берутся из токена пользователя |

> **Важно для фронтенда:**
> - `POST /admin/catalog/actualizations` **не принимает тело запроса** (или принимает `{}`).
> - `user_id` и `user_full_name` автоматически извлекаются из токена авторизации.
> - При создании задачи **автоматически** происходит импорт товаров из внешнего каталога.
> - Вся операция (создание задачи + импорт товаров) выполняется в **одной транзакции**. При ошибке импорта задача **не создаётся**.

### ActualizationTaskUpdateRequest — тело запроса для обновления задачи
| Поле | Тип | Обязательное | Описание |
|------|-----|-------------|----------|
| `status` | string | ❌ | Новый статус задачи (см. таблицу статусов ниже) |

> **Важно:** Все поля необязательны. Обновляются только переданные.

### ActualizationTaskResponse — ответ с данными задачи
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Уникальный ID задачи (формат `123e4567-e89b-12d3-a456-...`) |
| `user_id` | int | ID пользователя, создавшего задачу |
| `user_full_name` | string \| null | ФИО пользователя |
| `status` | string | Текущий статус задачи (см. таблицу ниже) |

### Статусы задач актуализации

| Статус | Описание |
|--------|----------|
| `CREATED` | Задача создана, импорт товаров выполнен успешно |
| `EXPORTING` | (зарезервировано) Процесс экспорта данных |
| `EXPORT_FAILED` | (зарезервировано) Ошибка экспорта из внешнего каталога |
| `IMPORT_FAILED` | (зарезервировано) Ошибка импорта товаров в БД |
| `COMPLETED` | (зарезервировано) Задача завершена |

> **Примечание:** На текущий момент при успешном создании задачи статус всегда `CREATED`. Остальные статусы зарезервированы для будущего функционала.

### PaginatedResponse — ответ списка
| Поле | Тип | Описание |
|------|-----|----------|
| `items` | ActualizationTaskResponse[] | Массив задач текущей страницы |
| `total` | int | Общее количество записей (без пагинации) |

### 4.1 GET `/admin/catalog/actualizations` — Список задач актуализации

**Permission:** `pricing_engine:view`

**Query-параметры:**

| Параметр | Тип | По умолчанию | Ограничения | Описание |
|----------|-----|-------------|-------------|----------|
| `status` | string | — | — | Фильтр по статусу задачи (`CREATED`, `COMPLETED`, и т.д.) |
| `user_id` | int | — | — | Фильтр по ID пользователя |
| `limit` | int | `100` | 1–1000 | Количество записей на странице |
| `offset` | int | `0` | ≥ 0 | Смещение от начала списка |

> **Примечание для фронтенда:**
> - Все фильтры **независимы** и могут комбинироваться.
> - Сортировка — по `created_at` (убывание, newest first).
> - Для пагинации используйте `limit` + `offset`. Общее количество записей — в поле `total`.

**Пример запроса:**
```
GET /admin/catalog/actualizations?limit=20&offset=0
```

**Пример ответа (200):**
```json
{
  "success": true,
  "data": {
    "items": [
      {
        "id": "123e4567-e89b-12d3-a456-426614174000",
        "user_id": 42,
        "user_full_name": "Иванов Иван Иванович",
        "status": "CREATED"
      }
    ],
    "total": 1
  }
}
```

**Пример ответа (пустой список, 200):**
```json
{
  "success": true,
  "data": {
    "items": [],
    "total": 0
  }
}
```

### 4.3 POST `/admin/catalog/actualizations` — Создать задачу актуализации

**Permission:** `pricing_engine:create`

**Тело запроса:** Пустое `{}` (или не передавать тело вообще).

> **Что происходит при создании задачи:**
> 1. Создаётся запись задачи актуализации (статус `CREATED`).
> 2. Сервер делает запрос к **внешнему каталогу** (`/product/admin/catalog/export`) с JWT-токеном пользователя.
> 3. Полученные товары сохраняются в БД, привязываясь к созданной задаче.
> 4. **Всё это — в одной транзакции.** Если на любом шаге ошибка — **задача НЕ создаётся** (полный rollback).

**Пример запроса:**
```
POST /admin/catalog/actualizations
Authorization: Bearer <token>
```

**Пример ответа (200):**
```json
{
  "success": true,
  "data": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "user_id": 42,
    "user_full_name": "Иванов Иван Иванович",
    "status": "CREATED"
  }
}
```

**Пример ответа (404 — внешний каталог не найден):**
```json
{
  "success": false,
  "error": {
    "code": "not_found",
    "message": "Эндпоинт каталога не найден",
    "details": {}
  }
}
```

**Пример ответа (409 — недостаточно прав во внешнем каталоге):**
```json
{
  "success": false,
  "error": {
    "code": "conflict",
    "message": "Доступ запрещён",
    "details": {}
  }
}
```

**Пример ответа (500 — не удалось импортировать товары):**
```json
{
  "success": false,
  "error": {
    "code": "service_error",
    "message": "Не удалось создать каталог товаров: ни один товар не был импортирован",
    "details": {}
  }
}
```

**Пример ответа (503 — внешний сервис недоступен):**
```json
{
  "success": false,
  "error": {
    "code": "service_unavailable",
    "message": "Не удалось подключиться к сервису каталога",
    "details": {}
  }
}
```

> **Примечание для фронтенда:**
> - Запрос может занимать **несколько секунд** (зависит от размера каталога и скорости внешнего API).
> - Таймаут внешнего API: 30 секунд.
> - При успешном создании задачи все товары уже импортированы и доступны через `GET /admin/catalog/products?actualization_task_id=...`.
> - При ошибке задача **не создаётся** — пользователю нужно попробовать снова.
> - JWT-токен пользователя автоматически передаётся во внешний каталог для авторизации.

## 6. Рекомендации для фронтенда

### 6.1 Страница списка задач актуализации (таблица)
- Используйте `GET /admin/catalog/actualizations` с `limit=50`.
- Реализуйте клиентскую пагинацию через `offset` и `total`.
- Фильтры:
  - `status` — выпадающий список (`CREATED`, `COMPLETED`, и т.д.).
  - `user_id` — поле для фильтрации по пользователю.
- Сортировка на серверке: newest first (по `created_at DESC`).
- В таблице отображайте: ID, ФИО пользователя, статус, дату создания.
- Клик по строке — переход к деталям задачи и списку товаров.

### 6.2 Создание задачи актуализации
- Кнопка «Создать задачу» → `POST /admin/catalog/actualizations` **без тела**.
- Показывайте **индикатор загрузки** — запрос может занимать до 30 секунд.
- При успехе — редирект на страницу задачи или показ toast «Задача создана, импортировано N товаров».
- При ошибке — показывайте понятное сообщение:
  - `503` — «Сервис каталога недоступен. Попробуйте позже.»
  - `404` — «Эндпоинт каталога не найден. Обратитесь к администратору.»
  - `500` — «Не удалось импортировать товары. Возможно, каталог пуст.»

### 6.3 Связь: Задача → Товары
- После создания задачи товары доступны через:
  ```
  GET /admin/catalog/products?actualization_task_id=<task_id>
  ```
- При удалении задачи — все товары удаляются каскадно. Показывайте подтверждение: «Удалить задачу и все привязанные товары?»

### 6.4 Обработка ошибок
| Код | Сообщение для пользователя |
|-----|---------------------------|
| `404` | «Задача не найдена» |
| `401`/`403` | Редирект на вход / «Нет прав» |
| `500` | «Не удалось импортировать каталог. Попробуйте позже.» |
| `503` | «Сервис каталога недоступен. Попробуйте позже.» |

## 7. Архитектурные детали (для контекста)

> Это информационный раздел. Фронтенд не обязан его знать.

- **CQRS:** Чтение (Queries) и запись (Commands) разделены.
- **UUID:** ID задач — UUID v4.
- **Единая транзакция:** Создание задачи + импорт товаров — в одном `UnitOfWork`. При ошибке — полный rollback.
- **Внешний каталог:** `POST` к задаче делает запрос к внешнему API: `GET /product/admin/catalog/export` с JWT-токеном пользователя.
- **Каскадное удаление:** `ON DELETE CASCADE` на FK `products.actualization_task_id → actualization_tasks.id`.
- **События:** При создании/обновлении/удалении публикуются доменные события через EventBus.
